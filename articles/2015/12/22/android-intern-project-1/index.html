RecyclerView 多类型 Item

APP的首页主要是一个列表，为了方便更新首页内容，给定了五六种 View 类型，然后按着服务器返回的数据来添加到列表里。

这个是比较easy的了，主要以下几点：

  1. 创建所需的几种不同的 ViewHolder
  2. 实现 RecyclerView.Adapter 的 getItemViewType() 方法
  3. 在 onCreateViewHolder() 方法里根据参数 viewType ，生成的holder的类型
  4. 在 onBindViewHolder() 方法里根据参数 position ，判断类型，然后强转holder类型，进行相关 View 的操作

Show me the code ：

public class HomePageAdapter extends RecyclerView.Adapter<RecyclerView.ViewHolder> {
    //定义类型常量
    private static final int TYPE_HEADER = 0;
    private static final int TYPE_ITEM = 1;

    //实现getItemViewType()方法
    @Override
    public int getItemViewType(int position) {
        //根据数据源的数据判断
        switch (dataList.get(position).getType()) {
            case "header":
                return TYPE_HEADER;
            case "item":
                return TYPE_ITEM;
            default:
                return TYPE_ITEM;
        }
    }
    //创建对应的Holder
    @Override
    public RecyclerView.ViewHolder onCreateViewHolder(ViewGroup parent, int viewType) {
        switch (viewType) {
            case TYPE_BANNER:
                return new HeaderHolder(
                        LayoutInflater.from(context)
                                .inflate(R.layout.header, parent, false));
                break;
            case TYPE_ONE_PLACE:
                return new ItemHolder(
                        LayoutInflater.from(context)
                                .inflate(R.layout.item, parent, false));
                break;
            default:
                return null;
        }
    }

    @Override
    public void onBindViewHolder(RecyclerView.ViewHolder holder, int position) {
        switch (getItemViewType(position)) {
            case TYPE_HEADER:
                ((HeaderHolder) holder).text.setText(dataList.get(position).getText());
                //do something ...
                break;
            case TYPE_ITEM:
                //do something ...
                break;
        }
    }

    //other code here ...

}

RecyclerView 和 SwipeRefreshLayout 下拉滑动冲突

这个状况就是，RecyclerView 还没有滑动到顶端的时候，下拉会直接触发刷新，而不是先上滑到顶部。

此处应有图片

通过 google 在 stackoverflow 查到的解决方案主要就是，通过判断 RecyclerView 是否到达顶部，然后以此设置 SwipeRefreshLayout 的 enable 属性，防止不恰当地触发刷新。

在 RecyclerView 中，通过 LinearLayoutManager 的 findFirstCompletelyVisibleItemPosition() 方法获得第一个完全可见的 item 的位置

    /**
     * 更新刷新控件的enable状态
     */
    private void updateRefreshStatus() {
        //根据文档，RecyclerView中没有可见的item时，pos为-1
        int pos = ((LinearLayoutManager) recyclerView.getLayoutManager())
                .findFirstCompletelyVisibleItemPosition();

        //判断RecyclerView是否滑到了顶部
        swipeLayout.setEnabled(pos <= 0);
    }

ViewPager 和 SwipeRefreshLayout 横向滑动冲突

此处有效果图

ViewPager 在 SwipeRefreshLayout 中横向滑动的时候，稍微有一点下滑的动作，就会触发刷新。

解决方案：重写 SwipeRefreshLayout 的 onInterceptTouchEvent() 方法，横向位移足够大时，返回false，将触摸事件传递给子 View，即 ViewPager ，以此解决冲突。

/**
 * Created by Bruce Zheng on 15-12-23.
 */
public class VerticalSwipeRefreshLayout extends SwipeRefreshLayout {
    private int touchSlop;//系统滑动常量
    private float lastX;

    public VerticalSwipeRefreshLayout(Context context, AttributeSet attrs) {
        super(context, attrs);
        touchSlop = ViewConfiguration.get(context).getScaledTouchSlop();
    }

    @Override
    public boolean onInterceptTouchEvent(MotionEvent event) {
        switch (event.getAction()) {
            case MotionEvent.ACTION_DOWN:
                lastX = event.getX();
                break;
            case MotionEvent.ACTION_MOVE:
                final float eventX = event.getX();
                float diff = Math.abs(eventX - lastX);

                if (diff > touchSlop + 60)//增大容差，x偏移足够大，false
                    return false;
        }
        return super.onInterceptTouchEvent(event);
    }

}

奇葩的 Tablayout 动画

后来，产品又提了个需求，要给 TabLayout 加 特技 动画，duang，效果如下：

此处应有图片

随着列表上滑，TabLayout 向左收缩，列表下拉，TabLayout 向右扩张。

TabLayout 是放在一个 AppBarLayout 里面的，那么就是 AppBarLayout 上滑的时候，对 TabLayout 进行处理，让它收缩就好了嘛。

AppBarLayout 可以添加 OnOffsetChangedListener 来监听滑动偏移量，回调函数 onOffsetChanged(AppBarLayout appBarLayout, int verticalOffset) 在滑动时会不断被调用，在此计算偏移的进度，然后给 TabLayout 设置相应的缩放程度。

其中小小的数学知识： 顶部 title 的高度为 TOPBAR_SIZE，这也是 AppBarLayout 上滑的距离。 最终 TabLayout 右端显示的图标大小为 ICON_SIZE，这也是 TabLayout 左移的距离。 两者只要移动的进度(百分比)相同，那么就会同步完成缩放。

即 offset / TOPBAR_SIZE = tabWidthOffset / ICON_SIZE

offset为 AppBarLayout 当前偏移量，根据上式就能算得当前 TabLayout 的宽度偏移量，然后给它的 LayoutParams 设置宽度。

由于 AppBarLayout 在滑动时会不断调用 onOffsetChanged() 方法，然后不断修改 TabLayout 的 LayoutParams， 性能消耗不小，故设置了两个触发条件，间隔16ms刷新一次，位移3px刷新一次。这样减少了不少设置 LayoutParams 的次数，而动画效果基本不受影响。

    appBarLayout.addOnOffsetChangedListener(new AppBarLayout.OnOffsetChangedListener() {
        private long lastTime = 0;//上一次触发时间
        private int lastOffset = 0;//上一次的offset

        private final int THRESHOLD_TIME = 16;//时间间隔阈值
        private final int THRESHOLD_PIX = 3;//像素偏移量阈值

        private final float ICON_SIZE = dp2px(36f);//tabLayout上的搜索Icon大小
        private final float TOPBAR_SIZE = dp2px(48f);//topBar 高度

        @Override
        public void onOffsetChanged(AppBarLayout appBarLayout, int verticalOffset) {
            //verticalOffset为负值，AppBarLayout未上滑收缩时，此参数为0

            //时间间隔大于16ms刷新一次，位移大于3px刷新一次
                if (System.currentTimeMillis() - lastTime < THRESHOLD_TIME
                        && Math.abs(verticalOffset - lastOffset) < THRESHOLD_PIX) return;
                lastTime = System.currentTimeMillis();
                lastOffset = verticalOffset;

                //收缩程度 (0 ~ 1.0)
                float ratio = -verticalOffset / TOPBAR_SIZE;
                //获取 TabLayout 的 LayoutParams
                FrameLayout.LayoutParams lp = (FrameLayout.LayoutParams) tabLayout.getLayoutParams();
                //宽度减小，即右边界左移
                lp.width = appBarLayout.getWidth() - (int) (ratio * ICON_SIZE);
                tabLayout.setLayoutParams(lp);
        }
    }
